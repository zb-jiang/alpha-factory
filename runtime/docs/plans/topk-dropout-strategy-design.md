# TopKDropout 策略业务场景设计

## 当前配置参数

```yaml
TopKDropout:
  buy_top_n: 40          # 买入候选池：因子得分排名前 40 的股票
  sell_drop_to: 200       # 卖出阈值：持仓股票跌出前 200 名则卖出
  holding_count: 20      # 目标持仓数：最终持有 20 只股票
  max_drop_per_day: 5    # 每次调仓最多卖出 5 只股票
  min_score_coverage: 0.90  # 因子分数保有率阈值（低于此值本调仓日不操作）
```

## 新增参数说明

### `min_score_coverage` - 因子分数保有率阈值

**含义**：调仓日因子分数非缺失的股票数占股票池总数的最低比例。

**计算**：`保有率 = 因子分数非缺失的股票数 / 股票池总数`

**作用**：

- 若保有率 < `min_score_coverage`，本调仓日不做任何操作（排名都做不了，后续动作没有意义）
- 若保有率 ≥ `min_score_coverage`，缺失因子的股票不参与排名

**示例**：股票池有100只股票，`min_score_coverage` = 0.90

- 90只以上有因子分数 → 正常操作
- 仅85只有因子分数（85/100=85% < 90%）→ 本调仓日不操作

## 核心业务逻辑澄清

### `sell_drop_to` 的含义

**原则：** 只有跌出前 `sell_drop_to` 名才卖出（宽幅缓冲，降低换手）。

例如：持仓股票排名在 21-40 名之间，不卖出；只有跌出前 40 名才卖出。

### 继续持有股票的权重管理

**原则：** 继续持有的股票保持当前权重不变，不做任何交易调整。

**背景**：调仓时，继续持有的股票因价格涨跌，当前权重可能偏离初始建仓时的等权（5%）。有两种处理方式：

- **方式A（全量等权再平衡）**：每个调仓日将所有持仓股票的权重都调整回等权，对继续持有的股票也产生"部分卖出"或"部分买入"的小额交易。
- **方式B（保留不动）**：继续持有的股票保持当前权重不变，只对卖出的股票清仓、对新买入的股票建仓。

**选择方式B的理由**：

- TopKDropout 的核心目标是**降低换手率**，方式A会产生大量不必要的小额交易，增加交易成本
- 继续持有的股票不做调整，意味着权重会随价格自然漂移（涨得多的权重变大，跌得多的权重变小），但这种漂移是有限的：
  - 涨得多的股票最终会因排名上升而保留或因排名回落被卖出
  - 跌得多的股票权重自然缩小，对组合影响有限
  - 新买入的股票每次都以等权建仓，提供稳定的权重锚定
- 方式B是"低换手策略"的标准做法，用轻微的权重漂移换取更低的交易成本

**示例**：

初始建仓买入20只，每只5%。一周后调仓：

| 股票 | 买入时权重 | 当前权重（因涨跌漂移） | 方式A | 方式B |
|------|-----------|---------------------|-------|-------|
| A（涨了） | 5% | 6% | 卖出一小部分调回5% | **不动** |
| B（跌了） | 5% | 4% | 买入一小部分调回5% | **不动** |
| C（跌出门槛） | 5% | 4.5% | 完全卖出 | 完全卖出 |
| D（新买入） | - | - | 买入5% | 买入约5% |

**结论**：方式B下，只有"完全卖出"和"完全买入"两种交易，不存在"部分卖出"和"部分买入"的调整交易。

## 已确认的业务逻辑

1. **因子分数保有率检查**：保有率 < `min_score_coverage` 时本调仓日不操作
2. **`sell_drop_to`** **含义：** 只有跌出前 `sell_drop_to` 名才卖出（宽幅缓冲，降低换手）。
3. **`max_drop_per_day`** **含义：** 每个调仓日最多卖出 `max_drop_per_day` 只，超出部分放弃卖出。
4. **持仓中因子缺失的处理**：缺失因子的股票无法判断排名，继续持仓
5. **继续持有股票的权重管理**：继续持有的股票保持当前权重不变，不做任何交易调整（方式B），只有"完全卖出"和"完全买入"两种交易，不存在"部分卖出"和"部分买入"的调整交易
6. 当上述卖出操作完成后，计算当前持仓和holding_count之间的差距，从buy_top_n中排除已在kept中的股票后，按排名从高到低等权买入，直到持仓数达到holding_count或候选用尽。
7. **停牌、涨停、跌停处理**：信号日（调仓日）当天不用关注停牌、涨停、跌停股票，因为信号日也不用交易，只用基于因子得分进行排名就行。信号日+1天用开盘价进行交易，停牌直接跳过(不延续)，买入跳过涨停（不延续）、卖出跳过跌停（后续每日用开盘价重试卖出直到下一个调仓日,重试卖出以开盘价执行，如果开盘时不跌停就可以卖出）。因为存在停牌、涨停、跌停导致交易失败的情况，所以每个调仓日之后持股数量和holding_count会有差距，这个是正常的

```yaml
Execution:
  suspend_action: skip          # 停牌处理：skip（跳过，不延续）
  limit_up_action: skip_buy     # 涨停处理：skip_buy（跳过买入，不延续）
  limit_down_action: delay_sell # 跌停处理：delay_sell（跳过卖出，后续每日重试直到下一个调仓日）
```
8. **调仓日独立性**：每个调仓日独立计算信号，不考虑前序调仓日的遗留交易（停牌、涨停、跌停而导致的交易失败）
9. 现金缓冲 ：保留cash_buffer_ratio的现金缓冲，防止因滑点导致资金不足，导致交易失败
10. **择时覆盖层与方式B的关系**：方式B（继续持有的股票保持当前权重不变）是**正常情况**下的规则。当开启择时功能后，择时作为独立的风控覆盖层，会对所有 target_weights 进行缩放（如 RISK STATE 下乘以 reduce_to=0.6），此时 kept 股票的 target_weight 变为 `reduce_to × 市值权重`，而实际持仓的 current_weight 仍为 `1.0 × 市值权重`，会触发 partial sell 将仓位降到目标水平。这是择时驱动的风控减仓，与方式B的"不调整"不矛盾——方式B说的是正常情况下不调整，择时是额外的风控层。
11. **RISK STATE下不renormalize**：当RISK STATE下股票级过滤（RSI）阻止了部分新买入股票时，被阻止股票的权重直接设为0，**不进行权重重新归一化（renormalize）**。原因是：RISK STATE的目的是降低总仓位到reduce_to水平，将被阻止股票的权重重新分配给其他股票会违背减仓意图。因此kept股票的target_weight保持为`原始权重 × reduce_to`，确保partial sell正确触发。
