# 因子挖掘主干流程升级工作计划

## 一、目标

当前系统已经具备一条可运行的 LLM 因子挖掘主干流程：

- 使用 `runtime/config/feature_pool.yaml` 作为基础特征白名单
- 对基础特征做体检与摘要
- 将摘要和约束喂给大语言模型生成候选因子
- 对候选公式做白名单校验
- 基于样本内 IC、Rank IC、ICIR、胜率和简化回测进行筛选
- 在多轮迭代中把上一轮结果反馈给 LLM

这条主干流程作为第一版已经成立，但还缺少更像“量化研究系统”的上下文与稳健性机制。本计划的目标是，在不推翻现有架构的前提下，分优先级补齐四类关键能力：

1. 训练流程从单一区间反复试错，升级为更稳健的滚动式验证
2. 给 LLM 增加客观的市场状态背景信息
3. 补充风格暴露与中性化控制
4. 统一标签定义与回测执行口径

---

## 二、现状概览

当前主干流程大致如下：

1. `step01_init_datasource.py`：初始化数据源并生成样本预览
2. `step02_build_feature_pool.py`：生成特征池清单
3. `step03_health_check.py`：对基础特征计算缺失率、相关性、稳定性等指标
4. `step04_build_summary.py`：构建 `llm_summary.json`
5. `step05_call_llm.py`：调用 LLM 生成候选因子
6. `step06_validate_factor.py`：做公式与字段白名单校验
7. `step07_eval_factor.py`：计算样本内因子统计指标
8. `step08_backtest.py`：按规则回测、过滤不达标因子
9. `step09_score.py`：综合打分并输出 Top 因子
10. `step10_iterate.py`：重复以上流程做多轮迭代

当前系统的优势在于“有约束、能闭环、可验证”，但主要缺口在于：

- LLM 没有看到当前训练期的市场状态背景
- 多轮迭代都在同一训练区间上反复优化，容易逐步贴合样本内
- 因子有效性还没有区分是 Alpha 还是风格暴露
- 标签周期和回测执行口径仍然存在进一步对齐空间

---

## 三、优先级一：训练主循环升级为滚动式验证

### 1. 目标

解决“在同一个训练区间里反复试错”的问题，减少样本内过拟合，使 LLM 挖出的因子更接近可迁移的研究成果。

### 2. 建议方案

将现有“单一训练区间 + 多轮迭代”改造为分层验证结构：

- Discovery：用于生成与初筛候选因子
- Validation：用于验证因子是否在相邻时期仍稳定
- Test：保留为最终样本外盲测，不参与调参

进一步的进阶版可以采用 rolling walk-forward：

- 训练窗口 1：生成候选因子
- 验证窗口 1：筛掉不稳定因子
- 训练窗口 2：重复以上流程
- 将多个窗口的结果汇总，保留跨窗口稳定的因子

### 3. 落地工作项

- 在 `env.yaml` 中增加训练切分配置
- 明确 discovery / validation / test 三段时间区间
- 新增窗口切分逻辑，支持静态切分与 rolling walk-forward 两种模式
- 将 LLM 生成与调优只放在 discovery 阶段
- 将 validation 阶段作为“只检验不调参”的约束层
- 汇总多个窗口的稳定性评分，形成跨窗口排名

### 4. 验收标准

- 同一个因子必须同时通过 discovery 与 validation，才允许进入最终候选池
- 最终 Top 因子需要记录其在不同窗口中的 IC、Rank IC、收益与胜率
- 系统能明确区分“样本内发现”与“样本外确认”

### 5. 预期收益

- 明显降低因子被训练区间“磨出来”的风险
- 让 LLM 反馈回路从“记住某段历史”变成“寻找跨阶段稳定逻辑”
- 为后续市场状态分析和风格暴露分析提供更可信的基础

---

## 四、优先级二：为 LLM 增加市场状态背景信息

### 1. 目标

让 LLM 在挖因子时，不只是看到特征统计表，还能知道当前训练数据处于什么市场环境，从而更有针对性地设计因子逻辑与失效假设。

### 2. 原则

不直接给“拍脑袋式”的牛市、熊市结论，而是先由系统计算一组客观状态变量，再将它们结构化总结成 market context。

### 3. 建议引入的状态维度

- 趋势状态：大盘 20 日、60 日、120 日收益率，均线位置，回撤情况
- 波动状态：指数波动率、横截面波动率、涨跌分化度
- 成交状态：成交量、成交额、换手率相对历史分位
- 风格状态：大盘相对小盘、低波相对高波、价值相对成长
- 广度状态：上涨家数占比、创新高占比、行业同步性

### 4. 建议输出形式

在健康检查与摘要之间增加一个“市场状态摘要”环节，输出例如：

- 当前训练区间整体偏上行 / 下行 / 震荡
- 当前阶段波动处于高 / 中 / 低水平
- 当前阶段成交处于活跃 / 一般 / 冷清
- 市场风格偏大盘 / 小盘，偏趋势 / 反转
- 与历史上若干区间存在相似性

### 5. 落地工作项

- 定义市场状态所需的原始数据和派生指标
- 在训练区间内计算状态变量与分位数
- 生成结构化的 `market_context.json`
- 在 `step05_call_llm.py` 中把市场状态摘要加入提示词
- 要求 LLM 在输出 `reason` 与 `risk` 时显式说明适用场景和失效场景

### 6. 验收标准

- 每轮因子挖掘都生成一份市场状态摘要
- LLM 输出的候选因子包含更明确的“适用环境”和“失效条件”
- 后续回测结果可按市场状态切片查看表现差异

### 7. 预期收益

- 提升因子设计的上下文相关性
- 减少“在震荡期挖趋势因子、在趋势期挖均值回复因子”这类错配
- 为后续做 regime-aware 因子库打基础

## 五、优先级三：补充风格暴露控制与中性化能力

### 1. 目标

避免把风格暴露误判为 Alpha，提升因子的可解释性与可迁移性。

### 2. 当前问题

很多因子表面上有效，实际可能只是：

- 小市值暴露
- 高换手暴露
- 高波动暴露
- 某个行业或某种市场风格暴露

如果不控制这些暴露，样本内结果容易虚高，样本外表现也更容易失效。

### 3. 建议方案

先做“识别”，后做“控制”，分两步推进：

第一阶段：

- 先识别每个因子与核心风格因子的相关性
- 给出因子暴露画像，而不是立刻强制中性化

第二阶段：

- 增加标准化、去极值、行业中性化、市值中性化等处理选项
- 将“原始因子”和“中性化后因子”做并行评估

### 4. 建议优先纳入的控制项

- 截面去极值：clip / winsorize
- 截面标准化：zscore
- 市值暴露控制：按 `market_cap` 做回归残差或分组中性化
- 行业暴露控制：按 `industry` 分组去均值或残差化
- 流动性暴露控制：按 `turnover`、`amount` 观察偏移

### 5. 落地工作项

- 定义风格基准变量集合
- 为每个候选因子增加暴露分析报告
- 在健康检查或回测阶段新增暴露相关输出
- 增加可选的中性化处理开关
- 比较“原始因子”和“中性化后因子”的指标差异

### 6. 验收标准

- 每个入围因子都能输出基本风格暴露说明
- 系统能识别“看似强但本质是风格代理”的因子
- 中性化后仍然有效的因子优先进入正式候选池

### 7. 预期收益

- 提升因子的真实 Alpha 含量
- 降低因子在风格切换时突然失效的概率
- 让回测结果更容易解释，也更接近专业量化研究口径

## 六、优先级四：统一标签定义与回测执行口径

### 1. 目标

确保“LLM 被要求预测的东西”和“系统实际回测执行验证的东西”尽可能一致，避免研究目标和交易验证目标之间出现错位。

### 2. 当前需要关注的问题

- `target_label` 与 `label_horizon` 的定义是否和回测持有期严格一致
- 标签收益是绝对收益还是超额收益
- 标签收益的起止价格口径，是否和交易执行价格口径一致
- 调仓频率、持有周期、换仓日选择，是否和标签周期一致

### 3. 建议方案

先建立一张“标签—执行口径对照表”，明确：

- 标签名称
- 预测窗口
- 收益计算价格口径
- 调仓频率
- 买入价 / 卖出价
- 是否对 benchmark 做超额化

随后将标签和回测规则做成一一对应关系，避免系统处于“标签是 5 日收益，回测却按另一套节奏交易”的模糊状态。

### 4. 落地工作项

- 梳理当前标签与回测实现的真实口径
- 补一份标签与调仓规则映射规范
- 明确 future_5d_return 对应的标准执行方式
- 为后续扩展 future_10d_return、future_excess_return 等标签预留结构
- 在 LLM 提示词里增加对标签口径的说明

### 5. 验收标准

- 任意一个标签都能明确回答“预测什么、从哪里买、持有多久、如何结算”
- 回测输出和标签定义不再存在解释歧义
- 未来新增标签时不需要临时靠人工理解补漏洞

### 6. 预期收益

- 提升研究指标与交易结果的一致性
- 减少“统计有效但回测逻辑解释不清”的情况
- 为后续扩展不同持有期因子打下统一标准

---

## 七、实施顺序建议

建议按以下顺序推进，而不是四项并行：

### 第一阶段：先补验证框架

- 完成优先级一
- 先把 discovery / validation / test 或 rolling walk-forward 建起来

原因：

- 如果验证框架不升级，后面无论加多少背景信息，仍然容易样本内过拟合

### 第二阶段：再补市场上下文

- 完成优先级二
- 把市场状态摘要作为新的输入层喂给 LLM

原因：

- 先有稳健验证，再有市场上下文，才能更清楚地判断“上下文到底有没有提升因子质量”

### 第三阶段：补风格控制

- 完成优先级三
- 先做暴露识别，再逐步推进中性化

原因：

- 暴露识别比直接全量中性化更稳妥，更容易看到因子本质

### 第四阶段：统一研究与交易口径

- 完成优先级四
- 将标签、持有期、回测规则文档化并系统化

原因：

- 这是研究走向可长期维护、可扩展的关键收尾动作

---

## 八、建议采用的交付节奏

### 版本 A：最小可落地版

- 增加 discovery / validation 双阶段
- 增加基础 market context 输出
- 保持当前回测和评分框架不大改

适合目标：

- 尽快把系统从“能跑”提升到“更可信”

### 版本 B：研究增强版

- 改为 rolling walk-forward
- 增加状态切片分析
- 增加风格暴露画像
- 初步加入中性化对照评估

适合目标：

- 建立一套更像正式研究平台的因子挖掘框架

### 版本 C：生产准备版

- 标签、回测、调仓规则完全对齐
- 增加不同股票池、不同市场状态下的稳定性要求
- 建立因子准入标准与淘汰标准

适合目标：

- 从研究系统走向可持续迭代的生产研究流程

---

## 九、最终建议

如果只看当前投入产出比，建议下一步优先做下面三件事：

1. 先把单一区间多轮试错改成 discovery / validation 或 rolling walk-forward
2. 再增加客观的市场状态摘要，让 LLM 带着上下文去挖因子
3. 然后补因子风格暴露画像，避免把风格当 Alpha

标签与回测口径统一虽然排在第四优先级，但不代表不重要，它更像是把整个系统从“研究原型”推向“长期可维护框架”的基础工程。

因此，对当前项目最合适的总体策略是：

- 不推翻现有主干
- 先补验证层
- 再补上下文层
- 再补解释层
- 最后统一研究与交易口径

这样改造的风险最小，收益也最清晰。
