# Step11 后接 Alphalens 分析方案草图

## 1. 目标

本方案的目标不是改造当前 `step07 -> step08 -> step09 -> step11` 主链路，也不是用 Alphalens 替代现有回测与评分体系，而是在 `step11` 完成后，对最终因子做一轮独立的深度研究分析，用于解释因子为什么有效、有效性是否稳定、风险是否集中，以及为下一轮因子挖掘提供反馈。

本方案的核心定位是：

- 作为 `step11` 之后的独立复盘步骤
- 不参与当前轮次的因子筛选与打分
- 不反向修改当前 OOS 结果
- 只为下一轮研究和人工判断提供依据

建议将该步骤命名为：

- `step11b_alphalens_report.py`

或：

- `post_step11_alphalens_analysis.py`

其中更推荐 `step11b_alphalens_report.py`，因为它和现有 Step 编号体系更一致，也能清楚表达“这是 Step11 的后置分析步骤，而不是主链路的一部分”。

---

## 2. 为什么不插在 Step07 和后续步骤中间

当前工程里，`step07`、`step08`、`step09`、`step11` 已经形成一条稳定的业务链路：

- `step07`：计算因子值与横截面统计指标
- `step08`：做真实组合回测
- `step09`：计算综合得分并选出优胜因子
- `step11`：做最终样本外盲测

如果在 `step07` 后强行插入一个 `step07b`，会有两个问题：

- 工程问题：打断现有稳定流程，增加链路复杂度和维护成本
- 业务问题：Alphalens 指标容易和当前 `IC / 回测 / 总分` 指标混在一起，导致责任边界不清晰

而放到 `step11` 之后则有明显优势：

- 不改当前主流程
- 不改变当前候选因子的晋级逻辑
- 不污染 OOS 评估
- 适合只分析少量最终因子，资源成本可控

因此，本方案判断：**将 Alphalens 放到 Step11 之后，比把它硬插入 Step07 和 Step08 之间更合理。**

---

## 3. 本方案的业务边界

这一步必须严格视为“复盘分析”，而不是“再筛选一轮”。

允许做的事情：

- 对最终因子做分层收益分析
- 看 IC 时序稳定性
- 看分位数组合收益是否单调
- 看换手是否过高
- 看行业维度是否存在过度集中
- 把分析结论沉淀为下一轮研究经验

不允许做的事情：

- 根据 Alphalens 结果重新替换本轮 `step11` 的最终因子
- 根据 OOS 分析结果回头再调本轮筛选阈值
- 用 Alphalens 输出覆盖现有 `step08/step09` 结果

原因很简单：`step11` 的定位是样本外盲测。如果在看完 OOS 结果之后又对本轮做二次筛选，相当于把 OOS 数据重新变成了调参依据，会污染盲测口径。

因此本方案的纪律是：

- `step11` 负责最终验证
- `step11b` 负责结果解释
- `step11b` 的结论只服务于下一轮因子研究

---

## 4. 这个方案希望回答哪些问题

在当前系统中，`step09` 和 `step11` 能告诉我们“哪个因子最终得分高、收益更好”，但不一定能充分解释下面这些问题：

- 最终因子是否真的具备稳定分层能力
- 收益是否主要来自极少数几个观测周
- 因子的高低分组收益是否有单调性
- 因子的表现是否只集中在某几个行业
- 因子是否高换手、是否存在信号跳跃过快的问题

Alphalens 比较适合回答上述问题。

因此，`step11b` 更像一个“因子研究显微镜”，不是交易引擎，也不是评分器。

---

## 5. 推荐分析对象

不建议一开始就对所有候选因子做 Alphalens 分析，因为：

- 计算量较大
- 报告噪声很多
- 真正有业务价值的通常是最后胜出的少数因子

建议第一阶段只分析下面两类对象之一：

- `outputs/backtest/top3_factors.json` 中的 Top3 最终因子
- `outputs/backtest/final_score.csv` 中前 3 到前 5 名因子

更推荐优先使用：

- `top3_factors.json`

原因是它已经是当前业务链路中明确认可的“最终候选集合”，和你现有的主流程口径最一致。

---

## 6. 推荐输入

### 6.1 因子列表来源

优先读取：

- `outputs/backtest/top3_factors.json`

可选读取：

- `outputs/backtest/final_score.csv`

用途：

- 确定要做 Alphalens 研究的最终因子集合

### 6.2 因子值来源

优先使用：

- `outputs/backtest/factor_values.parquet`

理由：

- 该文件已经由 `step07` 统一产出
- 其中包含 `factor_name`、`instrument`、`datetime`、`raw_score`、`score`
- `score` 已经经过你项目现有的预处理逻辑，和主链路口径更一致

建议：

- Alphalens 侧优先使用 `score`
- 不建议重新按公式再现场计算一遍，避免和当前系统预处理口径不一致

### 6.3 价格数据来源

价格数据应继续使用当前本地数据加载逻辑，而不是单独引入另一套数据通道。

也就是说：

- 仍然通过项目自己的 `load_raw_data(...)` 获取价格序列
- 避免因为额外接入外部取数链路导致价格口径漂移

### 6.4 分组信息来源

如果需要做 grouped analysis，可优先使用当前原始行情中的：

- `industry`

这对识别因子是否过度依赖单一行业、行业暴露是否过重很有帮助。

---

## 7. 必须和当前项目对齐的口径

这是本方案最重要的一部分。  
Alphalens 能不能接进来，不在于能不能跑通，而在于和你当前策略研究口径是否一致。

### 7.1 观测日口径

当前你的横截面分析周期是一周，因此：

- Alphalens 不应默认按全部交易日做分析
- 应仅保留当前项目定义的“周观测日 / 调仓日”

否则就会出现：

- 现有 `step07` 是周频评估
- Alphalens 却按日频分析

最终导致两个体系的指标不能直接互相印证。

### 7.2 Forward Return 口径

Alphalens 需要 forward returns。这里不能直接用默认的 `1D / 5D / 10D` 套路，而必须与当前项目的标签预测期保持一致。

也就是说，forward return 的定义应尽量对齐：

- 当前 `analysis_rule.yaml` 中的标签 horizon
- 当前 `build_label_series(...)` 的预测收益口径

如果这里不对齐，Alphalens 的 IC 和分层收益结论就会和当前项目的 `step07` 产生解释冲突。

### 7.3 股票池口径

应沿用当前项目正在使用的股票池配置，包括：

- 指数成分股
- ST 过滤
- 次新股过滤
- 动态成分股设置

不要在 Alphalens 阶段再单独使用另一套股票池，否则得到的 grouped analysis 和 quantile analysis 都会偏离主链路口径。

### 7.4 因子预处理口径

因子输入建议直接使用 `factor_values.parquet` 中的 `score` 列，而不是 `raw_score`。

原因：

- `score` 已体现你当前的标准化、去极值、中性化等预处理逻辑
- 与 `step08` 和 `step09` 保持一致

如果后续有对比研究需求，可以把 `raw_score` 作为可选补充分析对象，但不建议一开始就把两套口径混用。

---

## 8. 推荐输出

建议单独输出到：

- `runtime/outputs/alphalens/`

每个最终因子一个子目录，例如：

- `runtime/outputs/alphalens/factor_xxx/`

每个因子建议输出以下内容：

- 因子基础摘要
- IC 分析结果
- 分位数组合收益分析
- 因子换手分析
- 行业分组分析
- 图表或摘要表

如果考虑第一阶段先轻量落地，建议优先输出：

- 一个总览 `summary.json`
- 一个表格摘要 `summary.csv`
- 若干关键图表文件

这样即使后续不看完整图，也能先快速比较多个最终因子的研究特征。

---

## 9. 最值得保留的分析维度

结合你当前项目，我认为最值得优先保留的是以下 4 类分析。

### 9.1 分层收益分析

这是最核心的能力之一。主要看：

- 高分组是否持续优于低分组
- 收益是否具备大致单调性
- 头尾分组差异是否明显

这可以帮助识别一种常见伪因子：

- `Rank IC` 看起来还可以
- 但分组收益没有单调性
- 最终只是少量极端点在支撑结果

### 9.2 IC 时序分析

你当前已经有 `mean_rank_ic` 和 `rank_ic_ir`，但 Alphalens 的价值在于进一步看：

- IC 是否稳定
- 是否存在明显阶段性塌陷
- OOS 期间是否只有少数几周贡献了主要效果

### 9.3 Turnover Analysis

这里要特别强调：

- Alphalens 的 turnover 更偏“因子分位组成员变化率”
- 当前 `step08` 的 turnover 是“真实组合交易换手率”

两者不是一回事，但都很有价值。

推荐理解方式：

- `step08 turnover`：交易成本和策略执行压力
- `alphalens turnover`：信号稳定性和持股池变动速度

### 9.4 Grouped Analysis

如果使用 `industry` 分组，可以判断：

- 因子是否只在某些行业有效
- 是否存在显著行业偏置
- 最终收益是否本质上来自行业暴露而不是因子本身

这对 LLM 生成因子尤其重要，因为很多看似“公式有效”的因子，实际上只是行业轮动代理变量。

---

## 10. 不建议直接纳入 Step09 总分的指标

第一阶段不建议把 Alphalens 的输出直接并入 `step09` 总分。原因包括：

- Alphalens 与现有回测/评分链路的业务边界不同
- 其指标对参数和分组方式较敏感
- 你当前还没有完成与周频口径的稳定对齐验证

因此第一阶段应坚持：

- 只出研究报告
- 不改当前排名逻辑
- 不影响 Top3 生成

后续如果验证发现某些 Alphalens 指标长期稳定、有明确业务解释，再考虑是否抽取少量摘要指标作为“辅助参考项”，而不是强行并入主分数。

---

## 11. 推荐实施顺序

### 第一阶段：只做 `step11` 后复盘

目标：

- 先验证 Alphalens 在本项目中的研究价值
- 不动现有主流程
- 不改变当前业务结果

输入对象：

- 最终 Top3 因子

输出结果：

- 每个最终因子的 OOS 因子研究报告

### 第二阶段：验证口径一致性

重点验证：

- Alphalens 的 IC 结论是否和 `step07` 方向一致
- 分层收益与 `step08` 回测表现是否大体一致
- grouped analysis 是否能解释部分收益来源

如果发现大量冲突，需要先回头排查：

- 观测日口径是否对齐
- forward return 定义是否对齐
- 股票池是否对齐

### 第三阶段：沉淀为下一轮研究反馈

当第一、第二阶段稳定后，可以考虑把 Alphalens 的研究结论整理成：

- 因子质量标签
- 下一轮 LLM Prompt 反馈
- 人工审查结论摘要

例如：

- “该因子 OOS 单调性差，疑似依赖极端样本”
- “该因子主要暴露在金融行业，泛化性待观察”
- “该因子换手偏高，交易层面实现成本可能偏高”

---

## 12. 本方案的最终判断

针对当前项目结构，结论如下：

- 将 Alphalens 放在 `step11` 之后是合理的
- 它更适合作为最终因子的研究复盘工具
- 不适合第一阶段就插入 `step07 -> step08 -> step09` 主链
- 不适合第一阶段就参与最终总分计算

从价值定位看，它最适合扮演的角色是：

- 最终因子解释器
- 因子分层与稳定性诊断工具
- 下一轮研究的经验反馈模块

一句话总结：

**Alphalens 值得接，但应轻集成、后置集成、只做复盘，不做本轮决策。**

---

## 13. 下一步建议

如果后续继续推进，建议先做的不是代码，而是先把下面 4 件事明确下来：

- 分析对象是 Top3 还是 Top5
- OOS 分析只做 test 区间，还是同时补做 validation 区间对照
- forward return 具体对齐到哪个 horizon
- grouped analysis 是否先只做 `industry`

这 4 件事一旦明确，再进入代码实现会更稳，不容易出现“能跑，但结论口径不一致”的问题。
