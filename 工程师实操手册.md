## 2. 核心配置文件 (`env.yaml`)

所有的核心运行逻辑都在 `config/env.yaml` 中控制。**每次运行前，请务必检查此文件！**

关键配置项解读：
```yaml
# 核心运行模式（极其重要！）
# 'train': 样本内训练模式。大模型会根据表现不断迭代生成新因子。
# 'test': 样本外盲测模式。不再调用大模型，直接用训练集选出的 Top3 因子在全新时间段跑回测。
run_mode: train

# 样本内 (In-Sample) 训练时间段
train_start_date: 2022-10-01
train_end_date: 2025-12-31

# 样本外 (Out-of-Sample) 盲测时间段
test_start_date: 2026-01-01
test_end_date: 2026-04-10

# 样本内训练切分方式
training_workflow:
  mode: static_split
  static_split:
    discovery_start_date: 2022-10-01
    discovery_end_date: 2024-12-31
    validation_start_date: 2025-01-01
    validation_end_date: 2025-12-31
  walk_forward:
    discovery_window_months: 24
    validation_window_months: 6
    step_months: 6
    max_windows: 3

# 因子筛选严格标准 (Step 08 使用)
min_rank_ic_to_backtest: 0.01
min_rank_ic_ir_to_backtest: 0.1
min_positive_ic_ratio: 0.4
enable_direction_filter: false   # 是否额外要求 llm_direction 与 empirical_direction 一致
```

其中新增的 `training_workflow` 是这一版流程升级的关键配置：

- `mode: static_split`：把样本内训练集切成固定的两段。前一段是 `discovery`，允许大模型反复试错、生成新因子；后一段是 `validation`，只复验 discovery 里留下来的候选因子，不再重新调参或重新生成公式。
- `mode: walk_forward`：采用滚动窗口。系统会自动生成多个 `window_XX`，每个窗口都执行一次 `discovery -> validation`，最后再把所有窗口的结果汇总成跨窗口稳定性排名。
- `mode: disabled`：关闭切分，整段 `train_start_date ~ train_end_date` 都当成 discovery 使用。这更接近旧版单区间迭代模式，适合快速调试，但更容易过拟合。

按当前示例配置，系统实际跑的是：

- `discovery`: 2022-10-01 ~ 2024-12-31
- `validation`: 2025-01-01 ~ 2025-12-31

也就是说：先用 2022-2024 年末的数据让大模型挖因子，再用 2025 年的全新样本内时段做一次独立复验，最后才进入 2026 年样本外盲测。

## 3. 标准作业流程 (SOP)

整个系统的工作流被设计为**"清理环境 -> 样本内训练 -> 样本外盲测 -> 第三方平台因子检验"**四阶段模式。当前项目聚焦于因子挖掘、因子评估与因子检验。升级后的关键不再只是“跑几轮 iter”，而是明确区分：

- `window_XX`：第几个时间窗口
- `iter_YY`：当前窗口内 discovery 阶段的第几轮迭代
- `stepNN`：这一轮里具体执行的脚本

### 3.1 总体关系：window / iter / step 树状图

训练主流程的真实层级关系如下：

```text
train 主流程
├─ window_01
│  ├─ discovery
│  │  ├─ iter_01
│  │  │  ├─ step01 初始化数据源
│  │  │  ├─ step02 构建特征池
│  │  │  ├─ step03 特征体检
│  │  │  ├─ step04 构建摘要
│  │  │  ├─ step05 调用 LLM 生成因子
│  │  │  ├─ step06 校验因子公式
│  │  │  ├─ step07 因子评估
│  │  │  ├─ step08 策略回测
│  │  │  └─ step09 综合打分
│  │  ├─ iter_02
│  │  │  └─ step01 ~ step09
│  │  └─ iter_03
│  │     └─ step01 ~ step09
│  └─ validation
│     ├─ step01 初始化数据源
│     ├─ step02 构建特征池
│     ├─ step07 因子评估
│     ├─ step08 策略回测
│     └─ step09 综合打分
├─ window_02
│  ├─ discovery
│  │  ├─ iter_01
│  │  ├─ iter_02
│  │  └─ iter_03
│  └─ validation
└─ cross-window 汇总
   ├─ 汇总各 window 的 validation 全量结果
   ├─ 生成 cross_window_factor_ranking.csv
   └─ 生成 cross_window_top3_factors.json
```

这张图的含义非常重要：

- `window` 控制时间切片，是“在哪一段历史区间里做实验”
- `iter` 只存在于 `discovery` 阶段，是“在同一窗口里让系统连续试错几轮”
- `validation` 不再生成新因子，只验证 discovery 留下来的候选因子
- `cross-window` 汇总的是各窗口 validation 的**全量通过结果**，不是只汇总每个窗口的 Top3

### 3.2 阶段零：清理环境 (Clean Environment)

**目标**：在开始新一轮实验前清理运行态输出，避免旧结果污染本轮 discovery、validation 和 cross-window 汇总。

**操作步骤**：
1. 打开终端，进入 `runtime` 目录。
2. 先预览再执行：
   ```bash
   python src/step00_clean.py --dry-run
   python src/step00_clean.py
   ```

**清理范围**：
- `outputs/health/`：体检、摘要、预览文件
- `outputs/llm/`：LLM 原始输出、校验通过/拒绝结果
- `outputs/backtest/`：评估、回测、打分结果
- `outputs/iter_*/`：旧版单区间迭代归档
- `outputs/train_windows/`：window 级 discovery / validation 归档

**注意事项**：
- 清理脚本现在会默认删除 `outputs/train_windows/`，确保新一轮训练不会混入旧的 window / iter 归档
- 如果你希望保留历史实验结果，请先手动备份 `outputs/train_windows/`
- 如果 CSV 正在被 Excel 占用，清理脚本会跳过相关文件

### 3.3 阶段一：样本内训练与挖掘 (In-Sample Training)

**目标**：先在 discovery 区间内让大模型多轮试错挖因子，再在 validation 区间内复验 discovery 候选池，最后把多个窗口的 validation 结果做跨窗口稳定性汇总。

**操作步骤**：
1. 打开 `config/env.yaml`，确认：
   - `run_mode: train`
   - `train_start_date` / `train_end_date`
   - `training_workflow.mode`
   - `iteration_count`
2. 进入 `runtime` 目录。
3. 执行：
   ```bash
   python src/step10_iterate.py
   ```

**真实执行逻辑**：
- 第一步：根据 `training_workflow` 生成一个或多个 `window_XX`
- 第二步：在每个 `window` 内执行 `discovery`
- 第三步：`discovery` 按 `iteration_count` 循环执行 `step01 ~ step09`
- 第四步：系统收集当前 `window` 的所有 discovery 迭代结果，构造 validation 候选池
- 第五步：在同一 `window` 的 validation 区间只执行 `step01`、`step02`、`step07`、`step08`、`step09`
- 第六步：把所有窗口 validation 的结果汇总为跨窗口总表

**discovery 和 validation 的本质区别**：

```text
discovery
├─ 目标：生成并筛选新因子
├─ 会执行：step01 ~ step09
└─ 会调用 LLM：是

validation
├─ 目标：复验 discovery 候选因子
├─ 会执行：step01、step02、step07、step08、step09
└─ 会调用 LLM：否
```

**validation 到底拿 discovery 的什么结果**：

这里是本版流程最容易搞混、但也最重要的设计点：

- validation 的输入**不是** discovery 每个 iter 的 `top3`
- validation 的输入也**不是** discovery 所有生成过的原始因子
- validation 的输入是：当前 `window` 下，各个 `iter_YY/backtest/final_score.csv` 中所有进入最终打分表的因子，再结合对应 `iter_YY/llm/factors_validated.json` 取回公式详情

也就是说，validation 验证的是：

```text
当前 window 的 discovery 阶段
→ 每个 iter 的 final_score.csv 里的全部入围因子
→ 按 formula 字符串去重
→ 同公式重复时，保留 discovery_total_score 更高的版本
→ 形成 validation 候选池
```

这里的“去重”是工程层面的字符串去重，不是数学意义上的公式等价判断。也就是说，如果两个公式本质逻辑相同，但写法不同，当前版本仍可能把它们当作两个不同因子。

`top3_factors.json` 在 discovery 阶段的角色只是“本轮摘要”，方便人看结果；它不是 validation 候选池的主输入。

**训练阶段要重点看的文件分三层**：

- `outputs/train_windows/window_XX/discovery/iter_YY/`
  - 单轮 discovery 的完整归档
  - 适合排查某个因子是在哪一轮被提出、在哪一轮被淘汰
- `outputs/train_windows/window_XX/validation/`
  - 当前窗口的独立复验结果
  - 这里的 `final_score.csv` 是当前窗口 validation 的完整排名
  - 这里的 `top3_factors.json` 是当前窗口 validation 的前三名摘要
- `outputs/backtest/`
  - 这里是整个训练主流程最终写回的总表
  - `cross_window_factor_ranking.csv` 是跨窗口完整排名，不仅仅是 Top3
  - `cross_window_top3_factors.json` 才是跨窗口最终前三名

**训练阶段建议按这个顺序看结果**：

- 先看 `outputs/backtest/cross_window_factor_ranking.csv`
  - 这是所有 window 的 validation 因子按公式字符串去重后形成的跨窗口聚合总表，不要把它直接理解成“已经跨窗口胜出”的因子列表
  - 当前版本没有做公式等价归一化判断，所以“逻辑相同但写法不同”的公式，仍可能在表里以多个条目出现
  - 先看每个公式的 `windows_passed`、`window_pass_ratio`、`mean_validation_total_score`，判断它是只在个别窗口出现，还是在多个窗口都表现稳定
- 再看 `outputs/backtest/cross_window_top3_factors.json`
  - 这里的 Top3 才可以理解为跨所有 windows 竞争后选出来的最终优胜因子
- 再看 `outputs/backtest/cross_window_summary.json`
  - 看窗口数、validation 行数、聚合后因子数等总览
- 最后按需下钻到 `outputs/train_windows/window_XX/...`
  - 追溯某个公式是在哪个 window、哪个 iter 脱颖而出的

### 3.4 阶段二：样本外盲测 (Out-of-Sample Test)

**目标**：把训练阶段已经证明“较优”的少数候选因子，放到完全没见过的 test 区间再次评估，检验是否过拟合。

**操作步骤**：
1. 打开 `config/env.yaml`，确认 `run_mode: test`
2. 执行：
   ```bash
   python src/step11_oos_test.py
   ```

**执行逻辑**：
脚本会自动收集训练阶段的 Top3 摘要文件：

1. `outputs/backtest/cross_window_top3_factors.json`
2. `outputs/train_windows/*/validation/backtest/top3_factors.json`
3. `outputs/iter_*/backtest/top3_factors.json`

这里的“优先顺序”不是说后面的文件不读，而是说：

- 系统会按 `formula` 去重
- 同一个公式如果在多个层级都出现，谁先被读到，谁就保留
- 因此 `cross-window > validation > iter` 实际表示“重复公式冲突时的保留优先级”

**step11 为什么只拿 Top3，不拿全量 final_score**：

- validation 属于训练流程内部，需要尽量别漏掉潜在好因子，所以拿的是 discovery 的较大全集
- OOS 属于终局盲测，只应该检验训练阶段真正脱颖而出的少量精华因子，所以只拿 Top3 摘要做候选池

**样本外阶段不会做的事情**：
- 不运行 `step03`
- 不运行 `step04`
- 不运行 `step05`
- 不运行 `step06`
- 不再调用 LLM

**产出文件与解读**：
- `outputs/llm/factors_validated.json`
  - 样本外阶段的候选因子池
  - 本文件由训练阶段 Top3 摘要拼出来，模拟成“已校验因子输入”
- `outputs/backtest/final_score.csv`
  - OOS 阶段完整排名表
- `outputs/backtest/top3_factors.json`
  - OOS 阶段最终前三名
- `outputs/backtest/strategy_metrics.csv`
  - OOS 区间的收益、回撤、夏普等硬指标

### 阶段三：第三方平台因子检验 (Factor Validation)

**目标**：在米筐等第三方量化平台上验证因子的 IC、ICIR 等指标，确保因子逻辑在不同数据源下依然有效。

**操作步骤**：
1. 生成米筐因子代码：
   ```bash
   python ricequant/factor_converter.py
   ```
2. 登录米筐平台，进入"因子研究" → "因子检验"。
3. 复制生成的 `.py` 文件内容到因子编辑器。
4. 设置检验参数（时间范围、股票池、基准等）。
5. 点击"运行检验"查看 IC、ICIR 等指标。

**验证标准**：
- 米筐平台的 IC、ICIR 与 Qlib 结果差异 < 0.02
- 因子方向一致（正相关/负相关）

**产出文件**：
- `ricequant/converted_factors/{factor_name}.py`: 米筐因子代码

## 4. 各子步骤拆解 (单步调试指南)

如果你不想一键运行，或者某个环节报错需要排查，可以单独运行 `step00` 到 `step11`。本节按“输入文件 -> 执行动作 -> 输出文件 -> 文件作用”的顺序，说明每一步真实做了什么。

### 4.1 先记住三类关键文件

为了避免看文件时混淆，先记住下面三类：

- `factors_validated.json`
  - 含义：当前阶段准备送去做评估的因子列表
  - 作用：`step07` 的直接输入
- `final_score.csv`
  - 含义：当前阶段所有进入最终打分的因子的完整排名表
  - 作用：validation 候选池、cross-window 汇总、人工排查的核心来源
- `top3_factors.json`
  - 含义：当前阶段 `final_score.csv` 的前三名摘要
  - 作用：结果浏览、OOS 候选来源

一句话记忆：

```text
factors_validated.json = 候选输入池
final_score.csv        = 当前阶段完整排名表
top3_factors.json      = 当前阶段前三名摘要
```

### Step 00: 清理临时文件 (`step00_clean.py`)

- **重要输入文件**：无
- **执行动作**：删除当前运行态的中间输出，避免旧结果影响新一轮实验
- **重要输出文件**：无固定产物
- **会清理的目录**：
  - `outputs/health/`
  - `outputs/llm/`
  - `outputs/backtest/`
  - `outputs/iter_*/`
  - `outputs/train_windows/`
- **文件作用说明**：
  - 这一步不是为了生成结果，而是为了保证后续每个 step 读取的是本轮新鲜数据
  - 其中删除 `outputs/train_windows/` 是为了避免旧的 `window_XX/discovery/iter_YY/` 归档再次被新一轮 validation 扫描到

### Step 01: 初始化数据源 (`step01_init_datasource.py`)

- **重要输入文件**：
  - `config/env.yaml`
- **执行动作**：
  - 读取数据源类型、日期区间、股票池配置
  - 初始化 Qlib 或其他数据提供者
  - 拉取样本股票的预览行情
- **重要输出文件**：
  - `outputs/health/sample_preview.csv`
- **文件内容说明**：
  - 这是一个样本股票的前 50 行行情预览
  - 列通常包括 `open`、`close`、`high`、`low`、`volume`、`factor`
- **文件作用**：
  - 用来快速确认数据源是否连通、时间区间是否正确、行情字段是否可用

### Step 02: 构建特征池 (`step02_build_feature_pool.py`)

- **重要输入文件**：
  - `config/feature_pool.yaml`
- **执行动作**：
  - 读取允许使用的原始字段、基础特征、算子白名单
  - 生成标准化特征池清单
- **重要输出文件**：
  - `outputs/health/feature_pool_manifest.json`
- **文件内容说明**：
  - `raw_fields`：底层需要加载的原始行情字段
  - `base_features`：允许大模型引用的基础特征
  - `allowed_operators`：允许大模型使用的算子列表
- **文件作用**：
  - 这是后续 `step03`、`step05`、`step06`、`step07` 的共同配置基础

### Step 03: 特征体检 (`step03_health_check.py`)

- **重要输入文件**：
  - `config/env.yaml`
  - `config/feature_pool.yaml`
  - 底层加载的原始行情数据
- **执行动作**：
  - 根据特征池构造基础特征面板
  - 计算特征缺失率、分布统计、与标签的关系、特征间相关性
- **重要输出文件**：
  - `outputs/health/feature_stats.csv`
  - `outputs/health/feature_corr.csv`
  - `outputs/health/health_summary.json`
- **文件内容说明**：
  - `feature_stats.csv`：每个基础特征的质量统计
  - `feature_corr.csv`：特征间相关矩阵
  - `health_summary.json`：体检结果的摘要版 JSON
- **文件作用**：
  - 为 `step04` 提供摘要原料
  - 帮助判断特征池是否存在缺失过高、冗余过强等问题

### Step 04: 构建 LLM 摘要 (`step04_build_summary.py`)

- **重要输入文件**：
  - `outputs/health/feature_stats.csv`
  - `outputs/health/feature_corr.csv`
  - `outputs/backtest/top3_factors.json`（如果存在）
- **执行动作**：
  - 把统计表压缩成更适合大模型阅读的摘要 JSON
  - 如果存在上一轮 Top3，则把上一轮经验一并塞入摘要
- **重要输出文件**：
  - `outputs/health/llm_summary.json`
- **文件内容说明**：
  - 包含特征质量摘要、相关性摘要、目标标签信息
  - 可能附带 `previous_round_top_factors`
- **文件作用**：
  - 这是 `step05` 组装 Prompt 的核心素材

### Step 05: 调用大模型 (`step05_call_llm.py`)

- **重要输入文件**：
  - `config/env.yaml`
  - `config/feature_pool.yaml`
  - `config/backtest_rule.yaml`
  - `outputs/health/llm_summary.json`
  - `outputs/backtest/top3_factors.json`（如果存在）
  - `outputs/backtest/skipped_factors.json`（如果存在）
- **执行动作**：
  - 组装 Prompt，要求 LLM 输出新的因子候选
  - 在升级后流程里，这一步只允许在 `discovery` 阶段执行
  - 如果运行时上下文是 `validation`，脚本会直接拒绝执行
- **重要输出文件**：
  - `outputs/llm/raw_response.json`
- **文件内容说明**：
  - 大模型原始返回内容
  - 尚未经过任何公式合法性校验
- **文件作用**：
  - 是 `step06` 的唯一直接输入

### Step 06: 公式校验 (`step06_validate_factor.py`)

- **重要输入文件**：
  - `outputs/llm/raw_response.json`
  - `config/feature_pool.yaml`
  - `config/backtest_rule.yaml`
- **执行动作**：
  - 解析 LLM 返回的 JSON
  - 校验方向字段、特征白名单、算子白名单、公式合法性
  - 校验 LLM 是否私自篡改回测规则
- **重要输出文件**：
  - `outputs/llm/factors_validated.json`
  - `outputs/llm/factors_rejected.json`
- **文件内容说明**：
  - `factors_validated.json`：合法因子列表，方向字段已统一命名为 `llm_direction`
  - `factors_rejected.json`：非法因子及拒绝原因
- **文件作用**：
  - `factors_validated.json` 是 `step07` 的正式输入
  - `factors_rejected.json` 便于分析 LLM 为什么失败

### Step 07: 因子评估 (`step07_eval_factor.py`)

- **重要输入文件**：
  - `config/env.yaml`
  - `config/feature_pool.yaml`
  - `outputs/llm/factors_validated.json`
  - 原始行情数据
- **执行动作**：
  - 真正计算每个公式的 `raw_score`
  - 计算纯因子层面的指标，如 `mean_rank_ic`、`rank_ic_ir`、`positive_ic_ratio`
  - 推导样本内经验方向 `empirical_direction`
- **重要输出文件**：
  - `outputs/backtest/factor_values.parquet`
  - `outputs/backtest/factor_metrics.csv`
- **文件内容说明**：
  - `factor_values.parquet`：每个因子、每只股票、每个交易日的打分值
  - `factor_metrics.csv`：每个因子的纯因子指标和方向信息
- **文件作用**：
  - `factor_values.parquet` 是 `step08` 回测的直接输入
  - `factor_metrics.csv` 同时参与 `step08` 筛选和 `step09` 打分

### Step 08: 策略回测 (`step08_backtest.py`)

- **重要输入文件**：
  - `config/env.yaml`
  - `config/feature_pool.yaml`
  - `config/backtest_rule.yaml`
  - `outputs/backtest/factor_values.parquet`
  - `outputs/backtest/factor_metrics.csv`
- **执行动作**：
  - 根据阈值筛掉弱因子
  - 用 `empirical_direction` 决定是否反向使用分数
  - 对幸存因子执行组合构建、调仓、净值模拟
- **重要输出文件**：
  - `outputs/backtest/strategy_metrics.csv`
  - `outputs/backtest/skipped_factors.json`
  - `outputs/backtest/positions.parquet`
  - `outputs/backtest/orders.parquet`
  - `outputs/backtest/nav_curve.parquet`
- **文件内容说明**：
  - `strategy_metrics.csv`：幸存因子的策略收益指标
  - `skipped_factors.json`：被淘汰因子的名字和淘汰原因
  - `positions.parquet` / `orders.parquet` / `nav_curve.parquet`：回测明细
- **文件作用**：
  - `strategy_metrics.csv` 是 `step09` 的关键输入
  - `skipped_factors.json` 可以反哺下一轮 LLM 生成

### Step 09: 综合打分 (`step09_score.py`)

- **重要输入文件**：
  - `outputs/backtest/factor_metrics.csv`
  - `outputs/backtest/strategy_metrics.csv`
  - `outputs/llm/factors_validated.json`
- **执行动作**：
  - 合并纯因子指标和策略收益指标
  - 计算 `total_score`
  - 对高回撤、高换手、负收益因子施加惩罚
  - 排出完整排名并抽取 Top3
- **重要输出文件**：
  - `outputs/backtest/final_score.csv`
  - `outputs/backtest/top3_factors.json`
- **文件内容说明**：
  - `final_score.csv`：所有进入最终评分因子的完整排名表，不仅仅是 Top3
  - `top3_factors.json`：`final_score.csv` 的前三名，并补齐公式、方向、原因、风险说明
- **文件作用**：
  - `final_score.csv` 是 validation 候选池和 cross-window 汇总的核心来源
  - `top3_factors.json` 主要用于结果浏览和 OOS 候选来源

### Step 10: 训练主循环 (`step10_iterate.py`)

- **重要输入文件**：
  - `config/env.yaml`
- **执行动作**：
  - 根据 `training_workflow` 生成一个或多个 `window_XX`
  - 对每个窗口执行 discovery 多轮迭代
  - 从 discovery 中收集 validation 候选池
  - 对每个窗口执行 validation
  - 汇总所有窗口 validation 的结果，形成 cross-window 排名

**Step10 内部可以拆成三段理解**：

```text
A. 生成窗口
   └─ outputs/backtest/training_windows.json

B. 每个 window 执行 discovery
   └─ outputs/train_windows/window_XX/discovery/iter_YY/

C. 每个 window 执行 validation 并做 cross-window 汇总
   ├─ outputs/train_windows/window_XX/validation/
   └─ outputs/backtest/cross_window_*.*
```

- **discovery 阶段的重要输入/输出**：
  - 输入：运行时上下文里的 `window_id`、`stage=discovery`、`iteration`
  - 执行：`step01 ~ step09`
  - 输出：`outputs/train_windows/window_XX/discovery/iter_YY/` 完整归档
- **validation 候选池如何形成**：
  - 遍历 `outputs/train_windows/window_XX/discovery/iter_*/backtest/final_score.csv`
  - 读取对应 `iter_*/llm/factors_validated.json`
  - 用 `factor_name` 找回公式详情
  - 按 `formula` 去重
  - 同公式重复时，保留 `discovery_total_score` 更高的版本
- **validation 阶段的重要输入/输出**：
  - 输入：上一步构造出的候选池
  - 系统会先把候选池写入 `outputs/llm/factors_validated.json`
  - 然后执行 `step01`、`step02`、`step07`、`step08`、`step09`
  - 输出归档到 `outputs/train_windows/window_XX/validation/`
- **cross-window 汇总的重要输出文件**：
  - `outputs/backtest/cross_window_validation_details.csv`
    - 各窗口 validation 的明细表
    - 一行对应“某个 window 中某个公式的 validation 结果”
  - `outputs/backtest/cross_window_factor_ranking.csv`
    - 按 `formula` 聚合后的完整跨窗口排名表
    - 包含 `windows_passed`、`window_pass_ratio`、`mean_validation_total_score` 等字段
    - 这不是 Top3，而是完整总表
  - `outputs/backtest/cross_window_top3_factors.json`
    - `cross_window_factor_ranking.csv` 的前三名摘要
  - `outputs/backtest/cross_window_summary.json`
    - 窗口数量、validation 明细条数、聚合后因子数等总览

### Step 11: 样本外盲测 (`step11_oos_test.py`)

- **重要输入文件**：
  - `config/env.yaml` 中的 `run_mode: test`
  - `outputs/backtest/cross_window_top3_factors.json`
  - `outputs/train_windows/*/validation/backtest/top3_factors.json`
  - `outputs/iter_*/backtest/top3_factors.json`
- **执行动作**：
  - 从三个层级收集训练期 Top3 摘要
  - 按 `formula` 去重
  - 同公式重复时，保留先读取到的版本，因此实际优先级是 `cross-window > validation > iter`
  - 把收集结果写成新的 `outputs/llm/factors_validated.json`
  - 直接执行 `step07`、`step08`、`step09`
- **重要输出文件**：
  - `outputs/llm/factors_validated.json`
  - `outputs/backtest/final_score.csv`
  - `outputs/backtest/top3_factors.json`
  - `outputs/backtest/strategy_metrics.csv`
- **文件内容说明**：
  - 这里的 `factors_validated.json` 不再来自 LLM，而是来自训练期优秀因子的拼接结果
  - `final_score.csv` 是 OOS 的完整排名
  - `top3_factors.json` 是 OOS 的最终前三名
- **文件作用**：
  - 用最严格的 test 区间验证训练阶段选出来的少数精华因子是否真的稳健

## 5. 股票池管理指南

### 5.1 概述

`stock_pool_manager.py` 是系统的核心组件，提供统一的股票池管理能力。它支持多种股票池类型，确保在不同数据源（Qlib、米筐、聚宽等）下都能灵活配置股票池。

### 5.2 核心组件架构

```
stock_pool_manager.py  # 统一的股票池管理器
├── 支持多种股票池类型
│   ├── all_market: 全市场
│   ├── index_components: 指数成分股
│   ├── industry: 行业股票池
│   ├── custom_list: 自定义列表
│   ├── random_sample: 随机抽样
│   └── stratified_sample: 分层抽样
└── 跨平台兼容
    ├── Qlib: 当前数据源
    ├── 米筐: 未来数据源
    ├── 聚宽: 未来数据源
    └── Tushare: 未来数据源
```

### 5.3 配置方式

在 `env.yaml` 中配置股票池：

```yaml
stock_pool:
  type: "index_components"  # 股票池类型
  index_code: "000300.XSHG"  # 沪深300
```

更多配置示例参考 `runtime/config/env_stock_pool_examples.yaml`。

### 5.4 支持的股票池类型

#### 类型汇总

| 类型 | 说明 | 适用场景 | 支持状态 |
|------|------|----------|----------|
| `all_market` | 全市场股票池 | 全市场因子挖掘 | ✅ 完全支持 |
| `index_components` | 指数成分股 | 跨平台验证、生产环境 | ✅ 完全支持 |
| `industry` | 行业股票池 | 特定行业研究 | ⚠️ 需要数据源支持 |
| `custom_list` | 自定义股票列表 | 生产环境、特定研究 | ✅ 完全支持 |
| `random_sample` | 随机抽样 | 快速验证、开发调试 | ✅ 完全支持 |
| `stratified_sample` | 分层抽样 | 因子挖掘、保证代表性 | ✅ 完全支持 |

#### 1. 全市场股票池
```yaml
stock_pool:
  type: "all_market"
```
**适用场景**：全市场因子挖掘
**跨平台兼容**：所有平台都支持

#### 2. 指数成分股（推荐用于跨平台验证）
```yaml
stock_pool:
  type: "index_components"
  index_code: "000300.XSHG"  # 沪深300
```
**适用场景**：与米筐、聚宽保持一致
**跨平台兼容**：
- ✅ 米筐：`rq.index_components("000300.XSHG")`
- ✅ 聚宽：`get_index_stocks('000300.XSHG')`
- ✅ Tushare：`ts.get_index_stocks('000300.SH')`

**支持的指数代码**：
| 指数名称 | 米筐格式 | Qlib格式 | Qlib市场 |
|---------|----------|----------|----------|
| 沪深300 | `000300.XSHG` | `SH000300` | `csi300` |
| 中证500 | `000905.XSHG` | `SH000905` | `csi500` |
| 中证1000 | `000852.XSHG` | `SH000852` | `csi1000` |
| 上证50 | `000016.XSHG` | `SH000016` | `sse50` |
| 深证成指 | `399001.XSHE` | `SZ399001` | `szse` |
| 创业板指 | `399006.XSHE` | `SZ399006` | `growth` |

**注意**：Qlib 数据需要包含对应指数的成分股数据才能使用。默认数据通常包含 `csi300` 和 `csi500`。

## 9. 数据源切换指南（Qlib ↔ 米筐）

### 9.1 为什么需要切换数据源？

**Qlib 数据**：
- ✅ 免费开源
- ✅ 本地存储，访问速度快
- ❌ 数据截止到 2022/12/31，无法获取最新数据

**米筐 RQData**：
- ✅ 数据实时更新到最新交易日
- ✅ 支持更多数据类型（分钟线、财务数据等）
- ❌ 需要购买服务（有试用额度）
- ❌ 需要网络连接，有流量限制

### 9.2 快速切换步骤

#### 步骤1：安装米筐客户端

```bash
pip install rqdatac
```

### 9.3 数据源对比

| 功能 | Qlib | 米筐 RQData |
|------|------|-------------|
| **数据更新** | 截止到 2022/12/31 | 实时到最新交易日 |
| **股票列表** | `D.instruments()` | `rqdatac.all_instruments()` |
| **指数成分** | `D.instruments(market='csi300')` | `rqdatac.index_components()` |
| **行情数据** | `D.features()` | `rqdatac.get_price()` |
| **数据频率** | 日线 | 日线/分钟线 |
| **财务数据** | 有限支持 | 完整支持 |
| **费用** | 免费 | 需购买服务 |
| **流量限制** | 无 | 试用 1GB/天 |

### 9.4 常见问题

**Q: 米筐数据需要多少钱？**
A: 米筐提供试用账户（每日 1GB 流量），付费版本根据数据类型和流量配额定价，具体请访问 https://www.ricequant.com/welcome/rqdata

**Q: 可以部分使用米筐数据吗？**
A: 可以。系统支持按配置切换数据源，你可以：
- 开发和测试阶段使用 Qlib（免费）
- 生产环境使用米筐（最新数据）

**Q: 米筐和 Qlib 的数据格式有差异吗？**
A: 系统已自动处理格式转换：
- 股票代码格式统一（000001.XSHE）
- 数据索引格式统一（instrument, datetime）
- 字段命名统一（$open, $close 等）

**Q: 如何回退到 Qlib？**
A: 只需修改一行配置：
```yaml
data_source: qlib
```
无需修改任何业务代码。

### 9.5 技术实现说明

系统采用**策略模式**实现数据源切换：

```
runtime/src/data_provider/
├── base_provider.py      # 抽象基类
├── qlib_provider.py      # Qlib 实现
├── ricequant_provider.py # 米筐实现
└── provider_factory.py   # 工厂类
```

业务代码通过统一接口访问数据：
```python
from common import get_data_provider

provider = get_data_provider()
provider.initialize()

# 统一的接口，无需关心底层数据源
instruments = provider.get_instruments()
data = provider.get_price_data(...)
```
