## 3. 训练与盲测机制（业务口径）

### 3.1 Step10（训练主循环）

Step10 的业务含义是“海选 -> 复试 -> 总榜”：

- 海选：每个窗口先执行一次 `step03/step03b`，再按 `iteration_count` 跑多轮 `step04~step09`
- 复试：把该窗口 discovery 各轮 `final_score.csv` 的全量入围因子汇总
- 去重：按 `formula` 去重；同公式保留 `discovery_total_score` 更高者
- 复验：把去重后的候选送入 validation，运行 `step07`、`step08`、`step09`
- 总榜：汇总所有窗口 validation 结果，生成跨窗口排名

海选阶段硬筛（来自 `step08`）：

- `|mean_rank_ic| >= min_rank_ic_to_backtest`
- `|rank_ic_ir| >= min_rank_ic_ir_to_backtest`
- `win_rate >= min_positive_ic_ratio`
- 若开启 `enable_direction_filter`，还需 `llm_direction == empirical_direction`

跨窗口排序规则（Step10 总榜）：

1. `window_pass_ratio` 降序
2. `mean_validation_total_score` 降序
3. `mean_annualized_return` 降序

这三个字段含义：

- `window_pass_ratio = windows_passed / total_windows`
- `mean_validation_total_score`：各窗口 `validation_total_score` 均值（来自 `step09` 的 `total_score`）
- `mean_annualized_return`：各窗口年化收益均值（来自 `step08` 回测指标）

Step10 重点看 4 个文件：

- `outputs/backtest/cross_window_factor_ranking.csv`（跨窗口完整榜）
- `outputs/backtest/cross_window_top3_factors.json`（跨窗口 Top3 摘要）
- `outputs/backtest/cross_window_validation_details.csv`（逐窗口明细）
- `outputs/backtest/cross_window_summary.json`（汇总概览）

### 3.2 Step11（样本外盲测）

Step11 的业务含义是“最终上岗考试”：

- 不再生成新因子
- 只复用训练阶段优胜候选
- 在 test 区间做真正的样本外检验

Step11 候选来源（当前版本）：

- `outputs/backtest/cross_window_top3_factors.json`
- `outputs/train_windows/*/validation/backtest/top3_factors.json`

然后按 `formula` 去重，写入 `outputs/llm/factors_validated.json`，再直接跑：

- `step07`
- `step08`
- `step09`

Step11 重点看 3 个文件：

- `outputs/backtest/final_score.csv`（OOS 完整排名）
- `outputs/backtest/top3_factors.json`（OOS 最终 Top3）
- `outputs/backtest/strategy_metrics.csv`（收益/回撤/夏普等核心指标）

## 4. 单步调试指南（Step00/Step01 ~ Step11）

### Step00 清理环境（`step00_clean.py`）

- 输入：无
- 动作：清理 `outputs/health`、`outputs/llm`、`outputs/backtest`、`outputs/train_windows` 等运行产物
- 输出：无固定文件
- 作用：避免旧结果污染新一轮训练或盲测

### Step01 预缓存 Tushare（`step01_precache_tushare.py`）

- 输入：`config/analysis_rule.yaml`、`config/feature_pool*.yaml(估算特征 warmup 天数，向前扩展缓存窗口，避免后面首次跑特征时缺历史窗口)`
- 动作：按 train/test 区间并集计算缓存窗口，预拉取 `daily`、`adj_factor`、`daily_basic`，并触发静态字段缓存
- 输出：无固定文件（主要为数据源缓存命中提升）
- 作用：减少 Step10/Step11 首次执行时的数据等待时间

### Step02 特征体检（`step02_health_check.py`）

- 输入：`config/env.yaml`、`config/feature_pool.yaml`、原始行情
- 动作：构造特征面板，计算统计质量与相关性
- 输出：
  - `outputs/health/feature_stats.csv`
  - `outputs/health/feature_corr.csv`
  - `outputs/health/health_summary.json`
- 作用：提供特征质量诊断，供 Step04/Step05 使用

时间范围说明：

- 单独跑 Step03：使用当前配置区间（train 或 test，看yaml中run_mode到底是train还是test）
- 在 Step10 中跑 Step03：使用当前 `window` 对应区间（由运行上下文控制）

### Step03 市场环境刻画（`step03_market_context.py`）

- 输入：`config/env.yaml`、`config/feature_pool.yaml`、`config/market_context.yaml`、原始行情
- 动作：基于训练区间构建市场环境特征，计算趋势/波动/流动性/分化/广度/风格等状态标签与统计摘要
- 输出：`outputs/health/market_context.json`
- 作用：为 Step05 提供结构化市场环境上下文（`train_context`），用于约束因子生成与失效场景描述

时间范围说明：

- 单独跑 Step03：固定使用 `train_start_date ~ train_end_date`（脚本内部强制 `run_mode=train`）
- 在 Step10 中跑 Step03：同样使用当前 `window` 的训练子区间（由运行上下文覆盖 train 起止）

### Step04 构建摘要（`step04_build_summary.py`）

- 输入：`feature_stats.csv`、`feature_corr.csv`、可选 `outputs/backtest/top3_factors.json`
- 动作：汇总体检结果并附带上一轮 Top3（若存在）
- 输出：`outputs/health/llm_summary.json`
- 作用：Step05 Prompt 的核心输入

### Step05 调用 LLM（`step05_call_llm.py`）

- 输入：`env.yaml`、`feature_pool.yaml`、`backtest_rule.yaml`、`llm_summary.json`
- 动作：组装 Prompt 并调用大模型生成候选因子
- 输出：`outputs/llm/raw_response.json`
- 作用：Step06 的唯一直接输入

限制说明：

- 只允许在 `discovery` 阶段运行
- 若当前阶段是 `validation`，脚本会直接报错拒绝执行

### Step06 公式校验（`step06_validate_factor.py`）

- 输入：`outputs/llm/raw_response.json`、`feature_pool.yaml`、`backtest_rule.yaml`
- 动作：
  - 校验 JSON 结构
  - 校验 `direction` 取值
  - 校验特征/算子白名单
  - 校验 `fields` 与公式真实引用一致
  - 校验 `backtest_rule` 与固定规则一致
- 输出：
  - `outputs/llm/factors_validated.json`
  - `outputs/llm/factors_rejected.json`
- 作用：把“原始 LLM 输出”变成“可执行候选池”

### Step07 因子评估（`step07_eval_factor.py`）

- 输入：`factors_validated.json` + 行情 + 配置
- 动作：计算因子值与因子统计指标（IC、RankIC、IR、方向等）
- 输出：
  - `outputs/backtest/factor_values.parquet`
  - `outputs/backtest/factor_metrics.csv`
- 作用：为 Step08 回测与 Step09 打分提供基础数据

### Step08 回测与硬筛（`step08_backtest.py`）

- 输入：`factor_values.parquet`、`factor_metrics.csv`、`env.yaml`、`backtest_rule.yaml`
- 动作：
  - 先做硬筛（IC/IR/胜率/可选方向一致性）
  - 再对通过因子跑组合回测
- 输出：
  - `outputs/backtest/strategy_metrics.csv`
  - `outputs/backtest/skipped_factors.json`
  - `outputs/backtest/positions.parquet`
  - `outputs/backtest/orders.parquet`
  - `outputs/backtest/nav_curve.parquet`
- 作用：决定哪些因子进入 Step09 的综合评分

### Step09 综合打分（`step09_score.py`）

- 输入：`factor_metrics.csv`、`strategy_metrics.csv`、`factors_validated.json`
- 动作：计算 `total_score`，排序并抽取 Top3
- 输出：
  - `outputs/backtest/final_score.csv`
  - `outputs/backtest/top3_factors.json`
- 作用：
  - `final_score.csv` 是完整排名（不是只看前三）
  - `top3_factors.json` 是摘要，用于展示和后续候选来源


## 5. 股票池管理指南

### 5.1 概述

`stock_pool_manager.py` 是系统的股票池入口组件。  
当前版本数据源为 `tushare-only`，股票池逻辑都基于 `TushareProvider` 获取标的。

### 5.2 核心组件架构

```
stock_pool_manager.py  # 统一的股票池管理器
├── 支持多种股票池类型
│   ├── all_market: 全市场
│   ├── index_components: 指数成分股
│   ├── industry: 行业股票池
│   ├── custom_list: 自定义列表
│   ├── random_sample: 随机抽样
│   └── stratified_sample: 分层抽样
└── 当前数据源
    └── TushareProvider
```

### 5.3 配置方式

在 `env.yaml` 中配置股票池：

```yaml
stock_pool:
  type: "index_components"  # 股票池类型
  index_code: "000300.XSHG"  # 沪深300
```

常用附加过滤参数：

```yaml
stock_pool:
  include_st: false
  include_new_stock: false
  new_stock_days: 60
```

- `include_st`: 是否保留 ST 股票
- `include_new_stock`: 是否保留上市未满 `new_stock_days` 的新股

### 5.4 支持的股票池类型

#### 类型汇总

| 类型 | 说明 | 适用场景 | 支持状态 |
|------|------|----------|----------|
| `all_market` | 全市场股票池 | 全市场因子挖掘 | ✅ 可用 |
| `index_components` | 指数成分股 | 生产环境、风格控制 | ✅ 可用 |
| `industry` | 行业股票池 | 行业内研究 | ⚠️ 依赖数据源行业接口 |
| `custom_list` | 自定义股票列表 | 固定研究池 | ✅ 可用 |
| `random_sample` | 随机抽样 | 快速验证、开发调试 | ✅ 可用 |
| `stratified_sample` | 分层抽样 | 控制样本代表性 | ✅ 可用（简化实现） |

#### 1. 全市场股票池
```yaml
stock_pool:
  type: "all_market"
```
**适用场景**：全市场因子挖掘和基准实验

#### 2. 指数成分股（推荐）
```yaml
stock_pool:
  type: "index_components"
  index_code: "000300.XSHG"  # 沪深300
```
**适用场景**：策略落地时更稳定的可交易池

**常用指数代码（系统统一格式）**：
| 指数名称 | 配置值 |
|---------|--------|
| 沪深300 | `000300.XSHG` 或 `SH000300` |
| 中证500 | `000905.XSHG` 或 `SH000905` |
| 中证1000 | `000852.XSHG` 或 `SH000852` |
| 上证50 | `000016.XSHG` 或 `SH000016` |
| 深证成指 | `399001.XSHE` 或 `SZ399001` |
| 创业板指 | `399006.XSHE` 或 `SZ399006` |

系统会在数据提供层自动映射成 Tushare 接口需要的格式（如 `000300.SH`）。

#### 3. 行业股票池
```yaml
stock_pool:
  type: "industry"
  industry_name: "银行"
  industry_level: 1
```
**适用场景**：行业内选股研究  
**注意**：是否可用取决于数据源对行业分类接口的支持

#### 4. 自定义股票列表
```yaml
stock_pool:
  type: "custom_list"
  instruments:
    - "000001.XSHE"
    - "600000.XSHG"
```
**适用场景**：生产环境固定池、白名单池

#### 5. 随机抽样
```yaml
stock_pool:
  type: "random_sample"
  base_pool_type: "all_market"
  sample_size: 100
  seed: 42  # 固定种子，保证可复现
```
**适用场景**：快速验证系统性能和链路连通性

#### 6. 分层抽样
```yaml
stock_pool:
  type: "stratified_sample"
  base_pool_type: "all_market"
  sample_size: 500
  stratify_by: ["market_cap"]
  seed: 42
```
**适用场景**：保证样本代表性  
**注意**：当前实现是“简化分层”，默认按代码排序分层近似处理，不是严格市值分位分层

## 9. 数据源说明（当前为 tushare-only）

### 9.1 当前状态

- 当前代码仅注册了 `tushare` 数据提供者
- `env.yaml` 中 `data_source` 需配置为 `tushare`
- 若配置为其他值，系统会直接报错并停止

示例配置：

```yaml
data_source: tushare
tushare:
  api_key: 你的 Tushare Token
  data_cache_dir: D:/test/factor-factory/runtime/data/tushare_cache
```

### 9.2 常见问题

**Q: 可以切换到 Qlib 或米筐吗？**  
A: 当前主干代码不支持直接切换。若要启用，需要补齐对应 provider 实现并在工厂里注册。

**Q: 现在最容易出错的配置是什么？**  
A: `data_source` 不是 `tushare`，或者 `tushare.api_key` 缺失/无效。

**Q: Tushare 接口限流会影响流程吗？**  
A: 会。建议开启缓存目录并避免频繁全量重跑；必要时先缩小股票池做联调。

### 9.3 技术实现说明

数据提供层当前结构：

```text
runtime/src/data_provider/
├── base_provider.py      # 抽象基类
├── tushare_provider.py   # 当前唯一实现

```

```python

instruments = provider.get_instruments()
```
