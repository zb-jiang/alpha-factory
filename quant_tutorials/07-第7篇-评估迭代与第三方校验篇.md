# 第 7 篇：评估迭代与第三方校验篇 —— 如何证明你不是在自欺欺人？

## 课程简介

量化投资界有一个可怕的幽灵，叫做**“过拟合（Overfitting）”**。
只要你给计算机足够多的时间去组合特征，它总能找到一个在“过去 3 年”收益率高达 500% 的完美公式。但只要你拿这个公式去买明天的股票，大概率会亏得倾家荡产。

为了防止“自欺欺人”，本项目设计了一套极其严密的**“训练 -> 滚动验证 -> 盲测 -> 第三方平台交叉比对”**机制。这正是正规量化机构与业余玩家的分水岭。

---

## 7.1 样本内（Train）与样本外（Test）的隔离

要防止过拟合，最基础的操作就是把历史数据“切成两半”。
- **训练集（Train / 样本内）**：大模型看着这段数据写公式、调参数。（相当于平时做练习题）。
- **测试集（Test / 样本外盲测）**：这段数据大模型**绝对没见过**。拿定型后的公式在这段数据上跑，如果还能赚钱，说明是真本事！（相当于最终的高考）。

在 `analysis_rule.yaml` 中，我们这样配置：

```yaml
# analysis_rule.yaml
run_mode: train  # 告诉系统当前是在练习，还是在高考？(train 或 test)

# 练习题的时间段（过去 3.5 年）
train_start_date: 2020-04-01
train_end_date: 2021-03-31

# 高考的时间段（最新盲测区间，大模型没见过这段数据）
test_start_date: 2026-01-01
test_end_date: 2026-05-15
```

---

## 7.2 滚动交叉验证（Walk-Forward Validation）

如果“平时练习”只做一套题，偶然性太大了。
为了选出真正的“抗把子”因子，系统（`Step10_iterate.py`）使用了一种叫 **Walk-Forward（滚动向前）** 的高级机制。

```yaml
# analysis_rule.yaml
training_workflow:
  mode: walk_forward  # 采用滚动机制
  walk_forward:
    discovery_window_months: 24   # 海选期：24个月（大模型在这段区间写公式）
    validation_window_months: 6   # 复验期：6个月（拿着海选的公式，在这6个月测一下）
    step_months: 6                # 步长：往前挪6个月，再重复上面的动作！
    max_windows: 3                # 总共滚 3 个窗口
```

**Step10 业务流程图（“海选 $\rightarrow$ 复试 $\rightarrow$ 总榜”）：**

```mermaid
flowchart LR
    subgraph 窗口 1 (例如: 2018~2020)
        direction TB
        A1(Discovery: LLM挖出100个因子) --> B1(Validation: 筛出前3名)
    end
    
    subgraph 窗口 2 (向后滚半年: 2018.6~2020.6)
        direction TB
        A2(Discovery: LLM挖出100个因子) --> B2(Validation: 筛出前3名)
    end
    
    subgraph 窗口 3 (再滚半年...)

```

**只有在多个滚动窗口里都能活下来、排名靠前的因子，才是真正穿越牛熊的霸主！**

## 7.3 训练窗口选择器：独立前置辅助工具

细心的你可能会问：“那我该把 `train_start_date` 设在几几年呢？”
随便设可能效果不好，因为如果你打算在 2024 年实盘（震荡市），你拿 2014 年（杠杆大牛市）的数据去训练，显然是“刻舟求剑”。

为此，系统提供了一个独立脚本 `train_window_selector.py`（搭配 `selector.yaml`）。
**它的作用是**：算法会先计算“现在的市场状态向量”，然后去历史中寻找“最像现在的几个月”，最后给你出具一份报告，推荐你：“老板，用 2019~2020 年的数据做训练效果最好！”

## 7.4 第三方交叉验证（一）：Ricequant 米筐因子校验

大模型在本地挖出的因子，虽然经过了 `Step10` 的重重筛选，但我们怎么确定本地计算的 IC/IR 指标是准确的呢？会不会是因为我们本地的 Tushare 数据或者计算口径有问题？

为了彻底消除这个疑虑，本项目引入了 **Ricequant（米筐）因子校验** 环节。在进入最终的盲测前，我们利用 `runtime/ricequant/` 目录下的脚本：

1. **生成 Notebook 代码**：通过 `rq_factor_validator.py` 将本地胜出的因子公式，自动翻译成能在米筐平台上运行的 Python 脚本。
2. **云端比对**：将代码粘贴到米筐平台的 Jupyter Notebook 中运行。利用米筐权威的数据源和内置因子库，再次对这些因子进行横截面分析。
3. **差异排查**：如果发现本地 IC 是 0.05，而在米筐测出来只有 0.01，这就触发了预警。我们需要借助于本地的 Parquet 过程数据和米筐的明细数据做逐个对比，排查到底是成分股取错了，还是财报增速（如 `netprofit_yoy`）口径不同。

只有在本地和米筐平台上表现一致的因子，我们才敢放心地把它送入终极盲测。

## 7.5 第三方交叉验证（二）：导出聚宽实盘代码（Step14）

这是本项目**最硬核、最硬气**的设计！

我们在本地测出了策略回测收益率 50%。有没有可能是我们本地回测代码有 Bug？（比如偷偷偷看了明天的收盘价，这叫未来函数）。
如何自证清白？**去业内公认的第三方实盘回测平台（如 JoinQuant 聚宽）上跑一遍！**

`Step14_joinquant_export.py` 可以一键把我们本地的因子公式和回测参数，**翻译成可以直接在聚宽平台粘贴运行的 Python 代码**！

### 本地与聚宽对齐的“5 Why”排查法

如果你发现本地回测赚了 50%，聚宽只赚了 20%，千万不要凭空猜测，请按照以下顺序做逐日对齐：

1. **信号日对齐**：本地周一调仓，聚宽是不是也是周一？
2. **股票池对齐**：当天选出来的“前 20 名”股票名单是不是完全一样？
   - *已知差异*：聚宽可能拿不到当天的动态停牌信息，或者底层除权数据微调。
3. **交易量对齐**：本地买 100 手，聚宽是不是也买 100 手？
   - *已知差异*：本地代码严格按照 100 股的整数倍向下取整，聚宽内部某些机制可能不是严格向下取整。
4. **价格对齐**：都是按次日开盘价成交的吗？
5. **底层数据差异**：这是最常见的原因。比如 tushare 的单位是“千元+手”，米筐/聚宽是“元+股”。或者财务同比增速字段（如 `netprofit_yoy`）两边的口径不一致。

只要你能把每个调仓日的日志拉出来逐笔对比，就没有任何 Bug 能够隐藏！这也是量化工程师最核心的基本功。
